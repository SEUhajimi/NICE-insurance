# 与 Claude Code 协作开发指南

> 目标：减少 token 消耗，提高响应准确率，建立可持续的开发节奏。

---

## 一、会话管理

### 每次开新会话时先做这两件事

**1. 让 Claude 读 memory**
```
你读取一下 memory，看看上次进行到哪了
```
Claude 会从 `~/.claude/projects/.../memory/` 中恢复项目上下文，避免你重新解释背景。

**2. 告诉 Claude 本次目标**
```
今天我们要做 X，不需要动 Y 和 Z。
```
明确边界比宽泛描述更省 token，也能防止 Claude "顺手"改动你不想碰的地方。

---

## 二、任务拆分原则

### 一次只做一件事

**不好的写法：**
```
帮我把注册流程改一下，顺便优化一下 Service 层，然后补充单元测试
```

**好的写法：**
```
先改注册流程，把 CustomerRegisterRequest 的邮箱校验改为不区分大小写
```
拆分原因：Claude 每次处理的信息越少，错误越少；出错时也更容易定位。

### 大功能拆成三步走

```
步骤 1：先说"帮我分析 X 的现有实现，不要改代码"
步骤 2：确认理解后说"好，现在改 X"
步骤 3：改完后说"检查一下刚才的改动有没有遗漏"
```

---

## 三、指令精度技巧

### 给文件路径，不给文件名

**模糊（要搜索，消耗 token）：**
```
修改一下 Service 里的购保逻辑
```

**精确（直接定位）：**
```
修改 hjb-server/src/main/java/.../service/impl/CustomerPortalServiceImpl.java
第 76 行的 setCustType
```

### 说清楚"改什么"和"不改什么"

```
只改 purchasePolicy() 方法，不要动 makePayment()
```

### 给 Claude 看错误原文

遇到报错时，直接粘贴完整的错误信息比描述问题更有效：
```
运行后报这个错，帮我看看：
[粘贴完整 stack trace]
```

## 四、减少 Token 消耗的具体做法

### 用 `!` 前缀自己跑命令

在终端输入 `! <命令>` 可以直接在当前会话执行，结果会出现在对话里，不需要 Claude 帮你跑：
```
! git diff HEAD
! mvn test
! git log --oneline -5
```

### 避免让 Claude 读整个大文件

**浪费型：**
```
帮我读一下整个 CustomerPortalServiceImpl.java 然后告诉我能优化什么
```

**节省型：**
```
CustomerPortalServiceImpl.java 第 100-150 行的 makePayment 方法，
检查并发安全有没有问题
```

### 让 Claude "只回答，不改代码"

探索阶段先问，确认后再动手：
```
分析一下现在的 JWT 过期处理逻辑，不要改代码，给我讲清楚就行
```

## 五、Memory 系统的用法

Claude Code 有持久记忆，存在本地文件里，跨会话有效。

### 主动触发保存

当你说了某个重要决定、偏好或约束，可以直接要求记住：
```
记住：这个项目不用写注释，代码够清晰就行
记住：数据库字段用下划线，Java 字段用驼峰，MyBatis 自动映射
记住：我们暂时不做分页，等 MVP 完成后再加
```

### 记忆的四种类型

| 类型 | 适合存什么 | 示例 |
|------|-----------|------|
| `user` | 你的技术背景、偏好 | "熟悉 Java，不熟悉前端框架" |
| `feedback` | 对 Claude 行为的纠正 | "不要在方法上加注释" |
| `project` | 项目决策、待做事项 | "custType 更新逻辑暂未完成" |
| `reference` | 外部资源位置 | "SQL 脚本在 /sql 目录" |

## 六、CLAUDE.md — 项目级持久指令

在项目根目录创建 `CLAUDE.md`，Claude 每次启动都会自动读取，不占对话 token。

适合写进去的内容：
- 项目架构概述（3-5 句话）
- 编码规范（不写注释、命名规则等）
- 禁止操作（不能删的文件、不能改的接口）
- 常用命令（构建、测试、数据库初始化）

示例结构：
```markdown
# HJB Insurance — Claude 指令

## 项目简介
Spring Boot 3 + MyBatis + MySQL 保险系统，Maven 多模块。

## 编码规范
- 不写注释，用有意义的命名代替
- 所有响应用 Result<T> 包装
- 异常统一抛出自定义 Exception，由 GlobalExceptionHandler 处理

## 禁止操作
- 不改 application.yml 中的默认值
- 不删除 sql/ 目录下的脚本

## 常用命令
- 构建：mvn clean package -DskipTests
- 启动：java -jar hjb-server/target/*.jar
```

## 七、减少误操作的保护措施

### 在指令末尾加"确认再做"

```
帮我重构 AuthServiceImpl，但先告诉我你打算改哪些地方，确认后再动手
```

### 涉及数据库操作时明确说明

```
只改 Java 代码，不要动 SQL 脚本
```

### 让 Claude 做完后自检

```
改完之后检查一下有没有引入新的 null 风险或遗漏的方法调用
```

## 八、常见反模式（避免这样做）

| 反模式 | 问题 | 替代做法 |
|--------|------|---------|
| "帮我看看整个项目有什么问题" | 范围太大，结果泛泛 | "检查 CustomerPortalServiceImpl 的事务边界" |
| "你觉得这个功能怎么实现好？" | 开放性太强，浪费轮次 | "用 A 还是 B 方案实现，我倾向 A，你有什么反对意见吗" |
| 一次提多个不相关问题 | Claude 可能漏掉某些问题 | 每条问题单独一条消息 |
| 让 Claude 猜你想要什么 | 猜错了要返工 | 给出明确的输入输出期望 |
| 修改完不验证就继续 | 错误叠加难追溯 | 每次改完用 `! mvn test` 或手动验证 |

## 九、典型对话模板

### 开始新会话

```
读取 memory，看看我们上次进行到哪了。
今天要做：[具体目标]
不需要动：[不相关模块]
```

### 请求代码修改

```
修改 [文件路径]，[具体方法名]：
- 现在的问题：[描述]
- 期望行为：[描述]
- 不改其他地方
```

### 发现 bug

```
[粘贴错误信息]
出现在 [操作步骤]，
相关文件：[路径]，
帮我定位原因，先不改代码
```

### 结束会话前

```
把今天做的决定和未完成的事情保存到 memory
```

## 十、Token 效率速查

| 操作 | 消耗 | 建议 |
|------|------|------|
| 读取整个大文件 | 高 | 指定行号范围 |
| 模糊搜索（让 Claude 猜文件位置） | 高 | 直接给路径 |
| 连续多轮澄清 | 高 | 第一条消息就说清楚 |
| 让 Claude 写详细文档/注释 | 中 | 除非明确需要，否则跳过 |
| 让 Claude 自检已改的代码 | 低 | 值得做，节省后续返工成本 |
| 使用 `!` 自己跑命令 | 极低 | 多用 |
